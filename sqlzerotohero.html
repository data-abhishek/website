What is Data?

Data refers to raw facts, figures, symbols, or statistics collected together for analysis, reference, or processing. It has no meaning on its own until it is organized or interpreted. Data can exist in many forms — numbers, words, measurements, observations, descriptions, images, audio, and videos.

Data → Information → Knowledge → Wisdom

📊 Raw Data → 📈 Processed Information → 🧠 Applied Knowledge → 💡 Actionable Wisdom

Real-world Example:
  • Data: Temperature readings: 25°C, 30°C, 28°C
  • Information: Average temperature this week is 27.7°C
  • Knowledge: This temperature range is ideal for plant growth
  • Wisdom: Schedule watering based on temperature patterns

Different Types of Data

  • Structured data is organized neatly into rows and columns, usually stored in databases like SQL. It follows a fixed model, making it easy to search, retrieve, and analyze. Examples include customer databases and spreadsheets.
  • Unstructured data is data without a predefined format, like text files, images, videos, or social media posts.
  • Semi-structured data has some organizational structure but not as rigid as structured data. It often uses tags or markers (like in JSON or XML files) to separate data elements, making it easier to analyze than purely unstructured data.

What is a Database?

A database is an organized collection of data that can be easily accessed, managed, and updated. It allows you to store, retrieve, modify, and delete data efficiently. Databases are used to handle large amounts of information in a structured way. For example, a school might have a database to store student records, or a bank uses a database to manage customer accounts.

Types of Database

  • Relational Database
  • Non-Relational Database
  • Distributed Database
  • Object-oriented Database
  • Hierarchical Database

The Two Important Databases

Relational Database

A relational database is a type of database that stores data in tables, which consist of rows and columns. Each table represents a different entity (such as customers, products, or orders), and the columns represent the attributes of those entities. The rows store the actual data.
Relational databases use structured query language (SQL) to manage and manipulate the data. The relationships between tables are established using keys, where a primary key uniquely identifies each row in a table, and foreign keys link rows across different tables.

Non-Relational Database

A non-relational database (also known as NoSQL database) is a type of database that does not use the traditional table-based relational model for data storage. Instead, non-relational databases use a variety of data models that allow for more flexibility and scalability, making them particularly useful for handling large volumes of diverse, unstructured, or semi-structured data.

Difference Between Relational and Non-Relational Databases

Relational Database Non-Relational Database
Stores data in tables (rows & columns) Uses flexible data models (documents, key-value, graph, etc.)
Uses SQL for queries Uses various query languages (MongoDB, Cassandra, etc.)
Best for structured data and complex relationships Best for unstructured/semi-structured data and scalability
Examples: MySQL, PostgreSQL, Oracle Examples: MongoDB, Cassandra, Redis

History of Database

Before databases, data was often stored in flat files or manual records. These were simple text-based files or paper-based records that lacked structure and made it difficult to perform efficient searches, updates, or data management. As data grew, the need for more efficient ways of storing and retrieving it became apparent.

  • 1970: Edgar F. Codd introduced the relational model for databases. This model allowed easier querying and management of data using SQL, which became the standard for interacting with relational databases.
  • IBM's System R (1970s): One of the first implementations of the relational model, used to prove the viability of Codd's ideas. It led to the development of commercial relational database systems.
  • SQL (1970s): Developed to interact with relational databases, enabling users to query, insert, update, and delete data using a standard language. SQL soon became the industry standard for relational databases.

Disadvantages of File-Based System

Data is stored in flat files, often in formats like CSV, TXT, or custom formats. These systems don't have a predefined structure, and there is no central way to define relationships between data.

  • Data Integrity Issues: Data integrity can be difficult to maintain, as there are no built-in constraints. Without strict rules, data can be inconsistent, redundant, or error-prone.
  • Complex Querying: Querying in file-based systems is often done manually by searching through the file. Complex searches, joins, and data analysis require custom scripts or manual effort, which can be time-consuming and error-prone.
  • Data Redundancy: Data redundancy is common, as files may store the same information in multiple places, leading to inefficient storage and possible data inconsistencies.
  • Backup and Recovery: Backing up file-based systems involves copying the files, and recovery can be slow and cumbersome. There is no built-in support for data recovery in case of failure.

Database Management System (DBMS)

A Database Management System (DBMS) is software that facilitates the creation, management, and manipulation of databases. It acts as an intermediary between users and databases, enabling efficient storage, retrieval, and management of data. DBMSs are designed to provide a systematic and structured way to store data, ensuring that it is easily accessible, secure, and consistent.

  • Data Definition: A DBMS allows users to define the structure of the data, including the creation of tables, relationships, constraints, and indexes. This is usually done using a Data Definition Language (DDL).
  • Data Manipulation: The DBMS supports data manipulation operations such as inserting, updating, deleting, and retrieving data. This is typically done using a Data Manipulation Language (DML).
  • Security: DBMS enforce security measures, including access control mechanisms, to ensure that only authorized users can access or modify the data. Permissions are granted to users based on their roles.
  • Data Integrity: A DBMS ensures that the data is accurate and consistent through the use of constraints such as primary keys, foreign keys, and unique constraints. This prevents data anomalies like duplication and inconsistency.

SQL (Structured Query Language)

SQL (Structured Query Language) is a standard programming language used to manage and manipulate relational databases. It is used to interact with relational database management systems (RDBMS) such as MySQL, PostgreSQL, Oracle, Microsoft SQL Server, and many others. SQL allows users to perform various tasks related to database management, including data retrieval, insertion, updating, and deletion.

Data Definition Language (DDL)

  • CREATE: Defines a new database object (e.g., table)
  • ALTER: Modifies an existing database object
  • DROP: Deletes an existing database object

Data Manipulation Language (DML)

  • SELECT: Retrieves data from a database
  • INSERT: Adds new records to a table
  • UPDATE: Modifies existing data in a table
  • DELETE: Removes records from a table

Different Types of Keys in SQL

  • Primary Key: A primary key is a column or a combination of columns in a database table that uniquely identifies each row in that table. It is used to ensure that no two records have the same value in the primary key field.
  • Unique Key: A unique key is a constraint that ensures all values in a column or group of columns are distinct across the table and allows one null value.
  • Foreign Key: A foreign key is a column or set of columns in one table that refers to the primary key in another table, used to maintain referential integrity between the two tables.

Data Types in SQL

Data types in databases specify the type of data that a column can hold. They help ensure data consistency, integrity, and efficient storage and retrieval. Choosing the correct data type is essential for proper database design and performance.

Common Data Types used in SQL

  • INT: Stores whole numbers without decimals
  • FLOAT: Stores decimal values
  • VARCHAR(n): Stores variable-length string of maximum n size
  • DATETIME: Stores date and time combined
Software for Hands-on: SSMS - SQL Server Management Studio

Constraints in SQL

Constraints are rules enforced on data columns to ensure data integrity and accuracy.

  • NOT NULL: Ensures a column cannot have empty values
  • PRIMARY KEY: Uniquely identifies each record
  • FOREIGN KEY: Links two tables together
  • UNIQUE: Ensures all values in a column are different
  • CHECK: Ensures values meet a specific condition
  • DEFAULT: Sets a default value when no value is specified
Example with Constraints:
CREATE TABLE Students ( student_id INT PRIMARY KEY, name VARCHAR(100) NOT NULL, email VARCHAR(100) UNIQUE, age INT CHECK (age >= 18), enrollment_date DATE DEFAULT GETDATE() );

Clauses in SQL

SELECT

Used to retrieve data from one or more tables in a database.

SELECT column1, column2, ... FROM table_name;

FROM

Specifies the table from which to retrieve or delete data.

SELECT * FROM employees;

WHERE

Filters records that meet a specific condition.

SELECT * FROM table_name WHERE Age >= 18;

GROUP BY

Groups rows that have the same values in specified columns.

SELECT column, AGGREGATE_FUNCTION(column) FROM table GROUP BY column;

HAVING

Filters groups created by GROUP BY based on a condition.

SELECT column, AGGREGATE_FUNCTION(column) FROM table GROUP BY column HAVING condition;

ORDER BY

Sorts the result set by one or more columns.

SELECT column1, column2 FROM table ORDER BY column1 ASC|DESC;

LIMIT / TOP

LIMIT – Used in MySQL, PostgreSQL:

SELECT * FROM orders LIMIT 5;

TOP – Used in SQL Server:

SELECT TOP 5 * FROM orders;

SQL CASE Statements

The CASE statement in SQL is a versatile conditional expression that enables us to incorporate conditional logic directly within our queries. It allows you to return specific results based on certain conditions, enabling dynamic query outputs.

SELECT emp_id, salary, CASE WHEN salary > 100000 THEN 'High' WHEN salary BETWEEN 50000 AND 100000 THEN 'Medium' ELSE 'Low' END AS Salary_Level FROM employees;

Indexes in SQL

An index in SQL is a database object that improves the speed of data retrieval operations on a table, like an index in a book helps you quickly find a topic.

CREATE INDEX IX_EmpID ON Sudish_Info(Emp_Id);

Benefits of using Indexes in SQL:

  • Speeds up data retrieval using SELECT queries
  • Improves performance of WHERE, JOIN, ORDER BY, and GROUP BY clauses
  • Reduces disk I/O and CPU usage by narrowing data scans
  • Supports data integrity through UNIQUE indexes
  • Enhances sorting and filtering for reports and dashboards
  • Optimizes performance on large tables with millions of rows

Aggregate Functions

Aggregate functions are built-in SQL functions that process multiple rows of data and return a single summarized result. They are used to perform calculations such as totals, averages, maximum values, and counts, which are useful for reports, summaries, and analytics.

Common Aggregate Functions:

  • COUNT(): Counts the number of rows - COUNT(column_name)
  • SUM(): Adds up all the values in a numeric column - SUM(Salary)
  • AVG(): Calculates the average of numeric columns - AVG(Age)
  • MIN(): Finds the smallest value in the column - MIN(Salary)
  • MAX(): Finds the largest value in the column - MAX(Salary)

JOINS in SQL

A JOIN in SQL is used to combine rows from two or more tables based on a related column between them (usually a foreign key relationship). Joins allow you to query data across multiple tables as if they were one.

Sample Tables for JOIN Examples:

Employees Table

emp_id name dept_id
1 John Doe 101
2 Jane Smith 102
3 Bob Wilson NULL

Departments Table

dept_id dept_name
101 Engineering
102 Marketing
103 Sales

Types of Joins in SQL:

INNER JOIN

Returns only the rows where there is a match in both tables.

SELECT e.name, d.dept_name FROM Employees e INNER JOIN Departments d ON e.dept_id = d.dept_id;
Result: John Doe - Engineering, Jane Smith - Marketing

LEFT JOIN

Returns all rows from the left table and matched rows from the right. Unmatched right-side rows return NULLs.

SELECT e.name, d.dept_name FROM Employees e LEFT JOIN Departments d ON e.dept_id = d.dept_id;
Result: John Doe - Engineering, Jane Smith - Marketing, Bob Wilson - NULL

RIGHT JOIN

Returns all rows from the right table and matched rows from the left table. Unmatched left-side rows return NULLs.

SELECT e.name, d.dept_name FROM Employees e RIGHT JOIN Departments d ON e.dept_id = d.dept_id;
Result: John Doe - Engineering, Jane Smith - Marketing, NULL - Sales

FULL JOIN

Returns all records from both tables. If there is a match between the tables, it returns the matched rows.

SELECT e.name, d.dept_name FROM Employees e FULL JOIN Departments d ON e.dept_id = d.dept_id;
Result: John Doe - Engineering, Jane Smith - Marketing, Bob Wilson - NULL, NULL - Sales

Subqueries in SQL

A subquery is a query nested inside another query. It can be used in SELECT, INSERT, UPDATE, or DELETE statements.

Types of Subqueries:

Scalar Subquery (returns single value)

SELECT name, salary FROM employees WHERE salary > (SELECT AVG(salary) FROM employees);

Row Subquery (returns single row)

SELECT * FROM employees WHERE (dept_id, salary) = (SELECT dept_id, MAX(salary) FROM employees WHERE dept_id = 101);

Table Subquery (returns multiple rows)

SELECT name FROM employees WHERE dept_id IN (SELECT dept_id FROM departments WHERE location = 'New York');
💡 Performance Tip: Sometimes JOINs perform better than subqueries. Test both approaches for large datasets!

Window Functions

Window functions perform calculations across a set of rows related to the current row without grouping the result set.

Common Window Functions:

ROW_NUMBER()

SELECT name, salary, ROW_NUMBER() OVER (ORDER BY salary DESC) as rank FROM employees;

RANK() and DENSE_RANK()

SELECT name, salary, RANK() OVER (ORDER BY salary DESC) as rank, DENSE_RANK() OVER (ORDER BY salary DESC) as dense_rank FROM employees;

Partition By

SELECT name, dept_id, salary, AVG(salary) OVER (PARTITION BY dept_id) as dept_avg_salary FROM employees;
Real-world Use Case: Finding top 3 highest-paid employees in each department:
SELECT * FROM ( SELECT name, dept_id, salary, ROW_NUMBER() OVER (PARTITION BY dept_id ORDER BY salary DESC) as rn FROM employees ) ranked WHERE rn <= 3;

Database Design & Normalization

Database normalization is the process of organizing data to reduce redundancy and improve data integrity.

Normal Forms:

First Normal Form (1NF)

  • Each column contains atomic (indivisible) values
  • Each column contains values of the same type
  • Each column has a unique name
  • Order doesn't matter

Second Normal Form (2NF)

  • Must be in 1NF
  • No partial dependencies (non-key attributes depend on entire primary key)

Third Normal Form (3NF)

  • Must be in 2NF
  • No transitive dependencies (non-key attributes don't depend on other non-key attributes)
Normalization Example:

Views, Stored Procedures & Functions

Views

A view is a virtual table based on the result of an SQL statement.

CREATE VIEW high_salary_employees AS SELECT name, salary, dept_id FROM employees WHERE salary > 50000;
-- Using the view SELECT * FROM high_salary_employees WHERE dept_id = 101;

Stored Procedures

Stored procedures are prepared SQL code that can be saved and reused.

CREATE PROCEDURE GetEmployeesByDept(@dept_id INT) AS BEGIN SELECT name, salary FROM employees WHERE dept_id = @dept_id; END;
-- Execute stored procedure EXEC GetEmployeesByDept @dept_id = 101;

Functions

Functions return a single value and can be used in SQL expressions.

CREATE FUNCTION CalculateBonus(@salary DECIMAL(10,2)) RETURNS DECIMAL(10,2) AS BEGIN RETURN @salary * 0.10; END;
-- Using the function SELECT name, salary, dbo.CalculateBonus(salary) as bonus FROM employees;

Transactions & ACID Properties

A transaction is a sequence of operations performed as a single logical unit of work.

ACID Properties:

  • Atomicity: All operations succeed or all fail
  • Consistency: Database remains in valid state
  • Isolation: Concurrent transactions don't interfere
  • Durability: Committed changes are permanent

Transaction Control:

BEGIN TRANSACTION; UPDATE accounts SET balance = balance - 100 WHERE account_id = 1; UPDATE accounts SET balance = balance + 100 WHERE account_id = 2; -- If both updates successful COMMIT; -- If any error occurs -- ROLLBACK;
⚠️ Important: Always use transactions for operations that modify multiple related records!

Performance Optimization

Query Optimization Tips:

  • Use indexes wisely: Create indexes on frequently queried columns
  • Avoid SELECT *: Select only needed columns
  • Use LIMIT/TOP: Limit result sets when possible
  • Optimize JOIN conditions: Use proper indexes on join columns
  • Use EXISTS instead of IN: For better performance with subqueries
Example: Query Optimization
-- Slow query SELECT * FROM orders o, customers c WHERE o.customer_id = c.customer_id AND c.city = 'New York'; -- Optimized query SELECT o.order_id, o.order_date, c.name FROM orders o INNER JOIN customers c ON o.customer_id = c.customer_id WHERE c.city = 'New York' LIMIT 100;

-- Top 5 customers by total spending SELECT c.name, SUM(o.total_amount) as total_spent FROM customers c JOIN orders o ON c.customer_id = o.customer_id GROUP BY c.customer_id, c.name ORDER BY total_spent DESC LIMIT 5;
